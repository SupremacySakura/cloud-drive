# 学习日志目录 (docs/tasks/)

## 目录职责

本目录用于存放**学习日志**，记录任务执行的结果和教训，而非执行过程。

**核心定位**：
- 记录已完成任务的可复用经验
- 沉淀失败教训和解决方案
- 积累可复制的模式和方法论

**与 .sisyphus/plans/ 的关系**：
- `.sisyphus/plans/` 是执行计划目录，记录待办和进行中的任务
- `docs/tasks/` 是学习日志目录，记录已完成的反思和沉淀
- 两者分离：计划与执行 vs 结果与教训

## 任务文件格式规范

每个任务日志文件采用 **YAML frontmatter + Markdown body** 结构：

```yaml
---
task_id: <任务ID，与 .sisyphus/plans/ 中对应>
status: <completed | failed | cancelled>
created: <YYYY-MM-DD，创建日期>
last_reviewed: <YYYY-MM-MM，最后审查日期>
applicable_to: <适用场景描述，如 "所有 Node.js REST API 开发">
---

## 任务标题

## 结果

<任务执行的最终结果，量化指标如果适用>

## 教训

<从任务中学到的关键教训，包括失败原因分析>

## 可沉淀模式

<可复用到未来类似任务的模式、方法、最佳实践>
```

## 准入标准

进入本目录的任务文件必须满足：

1. **任务已完成**：无论成功或失败，任务必须有明确结论
2. **包含可复用教训**：至少有一条可复用的经验、模式或警示
3. **适用于未来类似任务**：教训能指导至少一个未来场景

**不进入本目录的内容**：
- 执行过程记录（应保留在代码库或会话历史中）
- 纯情绪性总结无实际指导价值的内容
- 已过期的临时方案

## 知识衰减策略

为确保知识有效性，实施以下策略：

- **审查周期**：每 6 个月审查一次所有任务日志
- **衰减标记**：如果 `last_reviewed` 超过 6 个月，文件顶部添加 `[待审]` 标记
- **过时处理**：
  - 仍有效的模式保留并更新 `last_reviewed`
  - 已过时的内容移至 `docs/tasks/archive/` 或删除
  - 技术栈变更导致不适用的内容标记为 `[已过时]`

## 文件命名约定

- 格式：`<简短描述>-<YYYYMMDD>.md`
- 示例：`nodejs-rest-api-lessons-20260430.md`
- 模板文件：`.template.md`（不以日期结尾）
