# AI 执行方式

## 来源

- 根目录 `AGENTS.md`（全局命令、架构、风险项）。
- 前后端 `AGENTS.md`（子域入口、命令、模块定位信息）。

## 适用范围

- 适用于 AI 或自动化代理在本仓库执行“分析、修改、验证、汇报”全流程任务。

## 更新入口

- 当仓库目录、启动命令、核心约束、验证流程发生变化时更新本文。
- 若引入新工具链（如测试框架、CI），需要同步新增对应执行策略与命令模板。

## 标准执行模板

1. 分析：确认需求边界，定位文件与受影响模块，识别跨文档同步点。
2. 修改：按最小必要改动原则落地，避免无关重构。
3. 验证：执行与改动匹配的最小验证命令，优先静态检查与关键路径回归。
4. 汇报：列出改动文件、核心行为变化、验证结果、残余风险与后续建议。

## 常用命令清单

- 依赖安装：`pnpm install`（仓库根目录）。
- 前端开发：`pnpm dev`（`apps/cloud-drive-frontend`）。
- 前端构建：`pnpm build`（`apps/cloud-drive-frontend`）。
- 根目录校验：`pnpm lint`、`pnpm typecheck`。
- 后端运行：`go run cmd/server/main.go`（`apps/cloud-drive-backend`）。
- 后端测试：`go test ./...`（`apps/cloud-drive-backend`）。

## 风险控制要点

- 包管理器统一使用 `pnpm`，避免混用 `npm`。
- 涉及 JWT、数据库 DSN、存储路径时，优先检查是否存在硬编码敏感信息。
- 文档改动需遵循“索引只导航、正文承载规则”的边界，不回填大段正文到索引页。
- 发现超出当前任务范围的大面积问题时，先汇报再扩展改动范围。

## 常见失败场景与处理

- 启动失败：先检查端口占用、数据库可连接性、环境变量是否缺失。
- 联调失败：先确认前端代理与后端端口匹配，再检查接口路径和鉴权头。
- 类型或 lint 失败：先修复当前改动直接引入的问题，再处理历史遗留项并单独标注。
- 回退策略：优先局部回滚当前提交范围内变更，避免影响无关功能。
