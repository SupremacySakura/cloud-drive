# Draft: 项目架构评估与后台管理需求分析

## 项目基本概况
- **类型**: 前后端分离的文件存储应用（网盘）
- **前端**: Vue 3 + TypeScript + Vite + Pinia + Tailwind CSS + Axios
- **后端**: Go + Gin + GORM + MySQL + JWT + Swaggo
- **部署**: Docker Compose（MySQL + Backend + Frontend），profiles 分层
- **核心功能**: 登录鉴权、文件上传与管理（分片上传/秒传/预览/下载）、分享功能（公共链接+取件码）

## 用户场景与需求
- **场景**: 企业内部网盘
- **管理范围**: 完整管理后台（仪表盘统计、用户管理、权限配额、系统配置等）

## 研究发现（已完成）

### 后端架构现状
- **分层**: router → handler → service → repository → database（标准清晰）
- **6 个模型**: UserModel / FileModel / FolderModel / UploadTask / PickUpCodeModel / PublicShareLinkModel
- **JWT**: 仅含 user_id，24h 过期，无角色信息
- **权限**: 极简！无角色/权限/组概念，所有登录用户完全平权
- **路由**: 21 个受保护路由 + 6 个公开路由，无管理员专属接口
- **用户功能缺失**: 无修改密码、无个人信息编辑、无头像上传
- **文件功能较完整**: 上传/下载/管理/分享/取件码/仪表盘
- **数据库迁移**: GORM AutoMigrate（无独立迁移脚本）

### 前端架构现状
- **9 个页面**: 首页/仪表盘/文件管理/上传/取件码管理/取件/登录/注册/需登录提示
- **Store**: 仅有 token 字段，无用户信息/角色
- **API**: auth（3个）+ file（18个），无管理类 API
- **路由守卫**: 仅判断是否登录，无角色/权限校验
- **无管理员路由/页面**

### 基础设施现状
- **Docker**: 4 个 profiles（infra/backend/frontend/full），健康检查完备
- **packages/**: 空目录，monorepo 结构预留但未使用
- **环境变量**: .env.example 完整，敏感信息硬编码需注意
- **数据库初始化**: 仅 bootstrap_records 表，其余由 GORM AutoMigrate 创建

### 关键缺失（与企业管理需求对比）
1. ❌ **无角色/权限体系** — 没有管理员概念
2. ❌ **无用户管理** — 无创建子用户、禁用、重置密码等
3. ❌ **无配额分配** — 虽然 UserModel 有 storage_limit，但管理员无法调整
4. ❌ **无系统监控** — 无系统日志、无操作审计
5. ❌ **无个人信息管理** — 无修改密码/编辑资料
6. ❌ **无组织架构** — 无部门/团队概念

## 技术决策
- **场景**: 企业内部网盘 → 需要多用户管理、配额控制
- **管理范围**: 完整管理后台 → 需要仪表盘/用户管理/权限/配额/日志/系统配置

## 开放问题（需确认）
1. 管理后台是独立前端还是嵌入现有前端？
2. 角色模型：简单 admin/user 两级，还是 RBAC 多角色？
3. 是否需要组织架构（部门/团队）？
4. 管理员创建用户的方式：仅后台创建，还是也允许邀请码？
5. 是否需要操作审计日志？
6. 系统配置需要哪些项？（存储路径、注册开关、配额默认值等）
